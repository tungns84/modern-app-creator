  <name>Tier-based PreToolUse enforcement decisions</name>
  <files>.claude/hooks/lib/tiers.mjs, .claude/hooks/t3-plan-gate.mjs, .claude/hooks/t4-command-guard.mjs, .claude/hooks/tests/*.test.mjs</files>
  <behavior>
    Decision function over (stdin payload, tiers.json, specs tree). Cases:
    - Write payload `{tool_name:"Write", tool_input:{file_path:".github/workflows/x.yml"}}` + no approved plan → deny JSON, reason contains "specs/" and "plan" and a next step
    - Same payload + approved plan present (plan.md with `Approved-by: <name>`) → no decision (exit 0, empty stdout)
    - Write to `.planning/notes.md` → no decision (meta exclusion, D-06)
    - Write to `src/whatever.ts` (no T3 match) → no decision
    - Bash payload `{tool_name:"Bash", tool_input:{command:"git push --force origin main"}}` → deny JSON, reason contains "human"
    - Bash payload `command:"task verify"` → no decision
    - Bash payload `command:"echo hacked > .cowork/tiers.json"` (write-bypass into T3 path) + no approved plan → deny JSON (best-effort scan: redirects `>`/`>>`, `tee`, `sed -i`, `git apply` targeting T3-matched paths)
    - Malformed stdin (not JSON) where script cannot prove path is safe → deny JSON containing "failing closed" guidance (Pitfall 1)
    - matchTier glob subset: `**` crosses dirs, `*` within segment, literal paths; `.cowork/**` matches `.cowork/tiers.json`; `**/pom.xml` matches `backend/pom.xml`
  </behavior>
  <implementation>
    Copy the fail-closed skeleton verbatim from 01-PATTERNS.md (deny() emitting hookSpecificOutput.permissionDecision deny + exit 0; whole script in try/catch; catch → deny for unprovable-safe inputs). Zero npm dependencies — only node:fs, node:path, node:child_process (git branch read). lib/tiers.mjs is the ONLY place glob matching lives (D-22); hand-rolled subset (**, *, literal), fallback to minimatch pre-approved only if globs outgrow the subset. Tests run scripts as child processes feeding fixture stdin and asserting stdout JSON — this tests the real contract, not internals.
  </implementation>
</feature>
