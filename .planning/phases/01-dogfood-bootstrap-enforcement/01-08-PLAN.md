    Write both ADRs with Status: Proposed and a full Alternatives-considered section each.

    ADR 0002 (FR-B11 — permission catalog code↔DB sync). Recommended decision: code is the source of truth — permissions declared as Java constants (one catalog class per module, `<module>.<resource>:<action>` codes per AUTHZ-01); a startup sync component reconciles the DB catalog table: inserts new codes, updates metadata (name/module/action/risk), marks codes missing from code as `status = DEPRECATED` (never deletes — deprecated-before-delete per AUTHZ-03); startup FAILS if a deprecated code is still referenced by any role grant after its grace window. Alternatives to document: (a) Flyway-migration-maintained catalog (rejected: drifts from code, manual), (b) annotation-processor build-time export + migration diff (rejected: heavier toolchain, same guarantees achievable at startup), (c) DB as source of truth (rejected: violates "no hard-coded roles BUT code-declared permissions" model and agent-readability). Consequences: deterministic catalog, testable sync job, Phase 3 implements the component + tests.

    ADR 0003 (FR-B13/GATE-11 — undeclared-permission detection). Recommended decision: two-layer static-first detection — primary: ArchUnit rule asserting every REST handler method (@RequestMapping family) carries `@RequirePermission` or an explicit `@PublicEndpoint` opt-out annotation (runs in `verify --fast`, names the violating method); backstop: a Spring context test enumerating `RequestMappingHandlerMapping` entries and asserting the same property at runtime wiring level (catches meta-annotation/edge cases ArchUnit misses). Alternatives: runtime-only enumeration test (rejected as sole layer: slower feedback, needs context boot), OpenAPI-spec diffing (rejected: detects after generation, indirect). Consequences: GATE-11 wires in Phase 3 as ArchUnit + context test; `@PublicEndpoint` becomes an auditable, greppable opt-out.

    Both ADRs: evidence and requirement texts summarized inline; zero `.planning/` links (D-07). Note in each: "Ratification: developer review at Phase 1 verification flips Status to Accepted."
