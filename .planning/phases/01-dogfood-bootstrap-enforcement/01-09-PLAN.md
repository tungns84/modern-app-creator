  <behavior>
    evaluate() over fixture inputs — one fixture pair (files, reviews) plus a branch value per case. Binding rule: head branch `feat/NNN-*` binds the PR to `specs/NNN-*/plan.md`; if the branch is unbound (no feat/NNN-* match), binding falls back to a `specs/NNN-*/plan.md` that is itself added/modified in changedFiles; OTHER approved specs in specsTree NEVER satisfy the gate. Cases (14):
    - Changed files all non-T3 (e.g., src/foo.ts, README.md) → PASS with reason "no T3 paths touched"
    - Changed files only under .planning/** → PASS (meta exclusion via shared matcher, D-06)
    - T3 file touched (.github/workflows/x.yml), branch feat/007-x, no specs/007-*/plan.md in specsTree → FAIL, reason names the touched path AND the expected bound artifact specs/007-*/plan.md
    - T3 touched, branch feat/007-x, specsTree contains ONLY unrelated approved specs (001..006 fixtures) → FAIL with reason "unrelated approved specs do not satisfy GATE-10" — the post-01-12 spec-rot case (checker blocker)
    - T3 touched, bound plan.md exists but lacks `tier: T3` or lacks `Approved-by:` → FAIL naming the missing marker
    - T3 touched, branch feat/002-x, specs/002-*/plan.md present+approved, reviews contain one APPROVED by login != author, user.type "User" → PASS
    - T3 touched, unbound branch (e.g., chore/cleanup), changedFiles includes specs/007-x/plan.md which is present+approved in specsTree, valid review → PASS (diff-binding fallback)
    - T3 touched, unbound branch, no plan.md in changedFiles, specsTree full of old approved specs → FAIL, reason instructs the feat/NNN-short-name convention
    - Reviews contain APPROVED then later CHANGES_REQUESTED by the same user → that user does NOT count (latest-per-user reduction, Pitfall 3)
    - Only review is APPROVED with user.type "Bot" → identity step fails (bot exclusion)
    - Only approval is by the PR author → identity step fails (no self-approval)
    - Identity step failed BUT waivers contain non-expired {scope:"self-approval"} → PASS with warning containing the waiver id (D-10) — the waiver bypasses ONLY the identity step, never the spec binding (a T3 change without its bound spec FAILS even under W-001)
    - Same waiver with expires < now → FAIL, reason "waiver expired" (waiver-rot, Security table)
    - reduceReviews: chronological list reduced to latest per login; only state APPROVED survives; DISMISSED never counts
  </behavior>
